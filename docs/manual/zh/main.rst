为什么要创建 Cabric 项目
--------------------------

作者早年是一名 PHPer,后来在转型 Python 中有幸认识 Fabric 这款部署工具。


作者的职业生涯一直走的是DevOPS路线，所以对于这样简单小巧的工具非常钟爱。（简单并不等于不安全，
有很多人对Fabric存在误解，认为是不安全的，在后面的章节我会讲到如何构造一个安全的部署方式）


严格的说，Fabric 并不是一个部署工具，他能做到的不仅仅是部署。
只需要加一小段代码，就可以完成期望的工作。



但Fabric问题也很明显，也就是因为需要加一小段代码，这部分代码仍然需要用户来实现。
在作者的眼中：

* 有实现就会有工作量
* 有实现就会有bug


可以预料的，当脚本累计到一定量的时，整个项目的运维工作其实是难以维护的。
而且这其中有很多脚本其实是使用频率相当高的，又有一些脚本其实只是一次性的。


所以作者提出一个设想，我们能不能基于Fabric，构造一个应用层。
用来解决：

* 简化最常用的工作的实现，不再需要定义Task来实现
* 与Fabric兼容，使之在一些复杂或特定的任务仍能继续使用Fabric


在作者管理的团队中，项目绝大多数都是基于CentOS实现的。于是`Cabric`这个项目就这样诞生了。


阅读前提
------------------

在开始学习使用Cabric之前，需要你具备如下技能：

* 懂的使用基本的shell指令
* 了解什么是json
* 了解Fabric的工作原理
